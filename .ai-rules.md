# AI Developer Rules & Standards

These rules are non-negotiable for all AI coding agents working on the Backend of this project.

## 1. Response Standardization

All backend API endpoints must return standardized JSON payloads matching the structure of `ResponseHttp`.

* **Standard Utility:** You MUST use the `ResponseHttp` utility class defined in [response.util.ts](file:///d:/WCAG22/job-matching-wcag22/apps/backend/src/common/utils/response.util.ts) for wrapping API responses.
* **Controller Handlers:**
  - Return `ResponseHttp.success(...)` explicitly from all controllers (or services) for all endpoints, specifying clear messages.
  - Never return raw objects or inline pagination objects formatted as `{ data: T[], total: number, page: number, limit: number }`.
* **Error Handling:**
  - Centralized exceptions are caught and formatted inside [all-exceptions.filter.ts](file:///d:/WCAG22/job-matching-wcag22/apps/backend/src/common/filters/all-exceptions.filter.ts).
  - Always map HTTP exceptions and validation errors using `ResponseHttp.error(...)`.

---

## 2. Password Hashing Security

Never use plain Bcrypt or standard hashing without pepper key.

* **Standard Utility:** You MUST use `HashUtil` defined in [hash.util.ts](file:///d:/WCAG22/job-matching-wcag22/apps/backend/src/common/utils/hash.util.ts).
* **Cryptographic Flow:** The utility implements double-hashing:
  1. Computes an **HMAC-SHA256** hash of the plain-text password using a secret Pepper key (loaded from `process.env.DB_PEPPER` with fallback to `xuantruong2004`).
  2. Hashes the output string using **bcryptjs** with `saltRounds = 12`.
* **Login & Auth Validation:** Always use `HashUtil.compare(password, hashedPassword)` for credentials validation.

---

## 3. Pagination Queries

All paginated endpoints (e.g. `GET` lists) must parse, validate, and bind parameters using a unified model.

* **Standard DTO:** You MUST use `BasePaginateQuery` from [base-paginate-query.dto.ts](file:///d:/WCAG22/job-matching-wcag22/apps/backend/src/common/dto/base-paginate-query.dto.ts).
* **Usage:** Pass the `@Query() query: BasePaginateQuery` object to controller endpoints and leverage it in services to perform query building, pagination limits, page offsets, and keyword searching.

---

## 4. Path Aliasing

Avoid using long relative paths like `../../` or `./../../` for imports.
* **Standard Alias:** You MUST use the `@/*` path alias mapping (defined in `tsconfig.json`) to refer to the `src/` directory.
* **Example:**
  - **Bad:** `import { EntityBase } from '../../../../common/entity/base.entity';`
  - **Good:** `import { EntityBase } from '@/common/entity/base.entity';`

---

## 5. Response Types

Never return raw TypeORM entities from service methods. Always use dedicated Response DTO classes.

* **Base Class:** All Response DTOs MUST extend `BaseResponse` from [base.response.ts](file:///d:/WCAG22/job-matching-wcag22/apps/backend/src/common/response/base.response.ts), which uses `@Exclude()` at class level and `@Expose()` on `id`, `createdAt`, `modifiedAt`.
* **Module-specific DTOs:** Each module MUST have a `response/` folder with a `.response.ts` file exporting a Response class (e.g., `UserResponse`, `JobResponse`).
* **Conversion:** Use `plainToInstance(XxxResponse, entity, { excludeExtraneousValues: true })` via a private `toResponse()` helper in each service.
* **Nested Relations:** Use `@Type(() => ChildResponse)` from `class-transformer` for nested relation fields.
* **Paginated:** Define `export type XxxPaginatedResponse = ResponseHttp<XxxResponse[]>;` in the same response file.
* **Example:**
  - **Bad:** `return this.userRepository.save(user);` (returns raw entity with `passwordHash`)
  - **Good:** `return this.toResponse(saved);` (returns exposed DTO structure)
