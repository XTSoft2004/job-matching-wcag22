# AI Developer Instructions: Accessibility Rules

## IMPORTANT NOTICE
This file contains strict, non-negotiable instructions for all AI coding agents working on this project. 

**Accessibility violations are bugs.**

Whenever generating React, Next.js, or HTML/CSS code, accessibility requirements take absolute precedence over visual design preferences or brevity.

## Generated Code MUST:
1. **Comply with WCAG 2.2 AA**: All code must meet these guidelines at a minimum.
2. **Use Semantic HTML**: Prefer native tags (`<button>`, `<nav>`, `<main>`, `<table>`) over `<div>` or `<span>`.
3. **Support Keyboard Navigation**: 
   - Never use `onClick` on a non-interactive element without `tabIndex` and `onKeyDown`.
   - Never create a keyboard trap.
4. **Support Screen Readers**:
   - Provide `aria-label` or visually hidden text (`sr-only`) for all icon-only buttons.
   - Use `aria-describedby` to link inputs to their error messages and helper texts.
   - Use `aria-live` for dynamic status updates.
5. **Include Labels**: Every `<input>`, `<textarea>`, and `<select>` MUST have a programmatically associated `<label>`.
6. **Preserve Focus Order**: Ensure tab order flows logically. Manage focus when opening/closing modals.
7. **Use Proper ARIA**: Follow the First Rule of ARIA. Only use ARIA when native HTML falls short.
8. **Pass Accessibility Testing**: Ensure generated code would pass `eslint-plugin-jsx-a11y` and axe-core checks without errors.

## Generated Code MUST NOT:
1. **Never Remove Focus Indicators**: Do not output `outline: none` or `focus:outline-none` unless you are simultaneously providing a highly visible alternative (e.g., `focus:ring-2`).
2. **Never Use Inaccessible Patterns**: Do not generate custom dropdowns, tabs, or modals without complete WAI-ARIA keyboard and state support.
3. **Never Rely on Color Alone**: Do not generate error states that only change the border color to red. Include text or icons.
4. **Never Skip Alt Text**: When generating `next/image` or `<img>`, always include an appropriate `alt` string (or `alt=""` for decorative images).

## Conflict Resolution
If a requested visual design or feature conflicts with accessibility requirements (e.g., "make this button invisible until I hover over it"), **always choose the accessible implementation** (e.g., ensure it is visible on focus as well). Accessibility is a non-negotiable engineering requirement.
